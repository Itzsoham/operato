Key design decisions to explain in interviews

  • Shared DB + restaurantId — simplest multi-tenant pattern, appropriate for early SaaS, easy to migrate to schema-per-tenant later. This same column generalises to tenantId when verticals arrive.
  • balanceAfter on InventoryTransaction — complete audit trail without summing all transactions on every read
  • orderNumber separate from id — UUID is for joins, "ORD-0042" is for staff and customer communication
  • totalSpend + visitCount denormalised on Customer — fast AI queries without slow aggregation joins
  • @@index on all date + status filters — shows you think about query performance from schema design
  • Generate-able relations (fixed) — Order now back-relates to Restaurant, Table and Customer (the original mislabeled Order.restaurant as a Table); Shift↔Staff and InventoryTransaction↔InventoryItem are wired on both sides. Without this, prisma generate fails.
  • restaurantId denormalised onto OrderItem, InventoryTransaction, Shift — these are exactly the tables the AI's marquee queries hit, so a flat WHERE restaurantId = $1 (+ RLS) keeps tenant isolation off the model's JOINs.
  • Postgres Row-Level Security — every tenant table has an RLS policy keyed on current_setting('app.restaurant_id'); even SELECT * with no WHERE returns only the active tenant. This is what actually makes text-to-SQL safe.
  • Transactional rollups — currentStock/balanceAfter and Customer.totalSpend/visitCount update inside a $transaction with SELECT … FOR UPDATE, so concurrent writes can't corrupt the audit trail.

AI Features

Three distinct features — each a separate engineering problem. Together they're what makes this project worth talking about, and the AI engine is the part that's fully vertical-agnostic.

1. Text-to-SQL Query Assistant

Owner types a business question. Gemini generates SQL, you execute it safely, Gemini streams back the analysis.

1
User types: "Which items sold most yesterday?"
2
Send to Gemini with schema context + restaurantId + instruction to return a parameterised SELECT only, as a typed object
3
Gemini returns a validated object { sql: "SELECT...", params: ["2024-01-15"] } via generateObject
4
Validate in code (reject writes, ;, comments) and enforce tenant isolation with Postgres Row-Level Security — never trust the model's WHERE clause
5
Execute on a read-only DB role inside a read-only transaction with statement_timeout + a forced LIMIT
6
Stream results back through Gemini for natural-language analysis — useChat() in the frontend renders tokens live
// src/app/api/restaurants/[restaurantId]/ai/query/route.ts
import { google } from '@ai-sdk/google'
import { generateObject, streamText } from 'ai'
import { z } from 'zod'

export async function POST(req, { params }) {
  const { restaurantId } = await params      // always from URL, never body
  await requireMember(restaurantId)          // EVERY route checks membership

  const { query } = await req.json()

  // Step 1: Generate SQL — structured output (not JSON.parse of free text)
  const { object } = await generateObject({
    model: google('gemini-2.5-flash'),
    schema: z.object({
      sql: z.string(),
      params: z.array(z.union([z.string(), z.number()]))
    }),
    system: buildSchemaContext(restaurantId) + SAFETY_RULES,
    prompt: query
  })

  // Step 2: Execute SAFELY — a SELECT-prefix check is NOT a sandbox.
  //   runReadonlySql uses a read-only DB role + read-only transaction +
  //   statement_timeout + forced LIMIT, rejects ; / comments / writes, and
  //   enforces tenant isolation via Postgres RLS (SET LOCAL app.restaurant_id) —
  //   so a missing WHERE clause from the model can never cross tenants.
  const data = await runReadonlySql(restaurantId, object.sql, object.params)

  // Step 3: Stream the analysis (serialize BigInt/Decimal before stringify)
  const stream = streamText({
    model: google('gemini-2.5-flash'),
    prompt: `Data: ${JSON.stringify(serializeRows(data))}\nQuestion: ${query}\nAnswer naturally.`
  })

  return stream.toUIMessageStreamResponse()
}
🔒 Why this is actually safe (the part the original plan got wrong)
Five layers, none of them "the prompt": (1) generateObject + Zod returns a typed { sql, params } — no JSON.parse of free text; (2) a static pre-filter rejects ;, comments and any write keyword; (3) execution runs on a dedicated read-only Postgres role inside a transaction_read_only transaction with a statement_timeout and a forced LIMIT; (4) Row-Level Security (SET LOCAL app.restaurant_id) means even a missing WHERE can't cross tenants; (5) results pass through serializeRows() to convert BigInt/Decimal before JSON.stringify — otherwise every revenue/count query throws. A startsWith('SELECT') check is none of these.

// src/app/api/cron/weekly-summary/route.ts  (throttle: shared Gemini key ~15 req/min)
import { google } from '@ai-sdk/google'
import { generateText } from 'ai'

const restaurants = await prisma.restaurant.findMany()

for (const r of restaurants) {
  const metrics = {
    revenue: await getWeekRevenue(r.id, weekStart, weekEnd),
    topItems: await getTopItems(r.id, weekStart, weekEnd, 5),
    newCustomers: await getNewCustomers(r.id, weekStart, weekEnd),
    orderCount: await getOrderCount(r.id, weekStart, weekEnd),
    busiestDay: await getBusiestDay(r.id, weekStart, weekEnd),
    lowStockItems: await getLowStock(r.id),
  }

  const { text } = await generateText({
    model: google('gemini-2.5-flash-lite'),  // cron: more free-tier headroom
    prompt: `Write a brief, friendly weekly summary for a restaurant owner.
    Data: ${JSON.stringify(metrics)}`
  })

  await prisma.weeklySummary.upsert({
    where: { restaurantId_weekStart: { restaurantId: r.id, weekStart } },
    create: { restaurantId: r.id, weekStart, weekEnd, summary: text, metrics },
    update: { summary: text, metrics }
  })
}

3. Smart Inventory Alerts

Not just "stock is low" — calculates sales velocity and gives a specific reorder recommendation.

// src/lib/ai/inventory-alerts.ts
import { google } from '@ai-sdk/google'
import { generateText } from 'ai'

// Calculate how many units sold per day (avg over last 7 days)
const salesVelocity = await getDailyUsageByItem(restaurantId, 7)

const urgentItems = inventoryItems
  .map(item => ({
    ...item,
    daysLeft: item.currentStock / (salesVelocity[item.id] || 0.1),
    weeklyUsage: (salesVelocity[item.id] || 0) * 7
  }))
  .filter(item => item.daysLeft < 4)  // less than 4 days
  .sort((a, b) => a.daysLeft - b.daysLeft)

const { text } = await generateText({
  model: google('gemini-2.5-flash'),
  prompt: `Based on sales velocity, suggest reorder quantities.
  Items: ${JSON.stringify(urgentItems.map(i => ({
    name: i.name, current: i.currentStock, unit: i.unit,
    daysLeft: Math.round(i.daysLeft * 10) / 10,
    weeklyUsage: Math.round(i.weeklyUsage * 100) / 100
  })))}`
})

Key architecture notes

  • Route Groups — (auth) and (marketing) share no layout with dashboard, clean separation
  • All dashboard routes are under [restaurantId] — but the ownership check (requireMember) runs in both layout.tsx (pages) AND every API route handler, since layouts don't run for /api routes
  • schema-context.ts builds a natural-language description of your DB for Gemini — generated from the Prisma DMMF so it can't drift, not the raw schema (too verbose, wastes tokens). This is the file a new vertical overrides.
  • lib/verticals/ is the seam for the platform play — each industry is a config pack (entities, KPIs, AI examples, terminology), not a forked app
  • lib/ai/ is completely separate from components — AI logic is testable independently of React
  • seed.ts is critical for portfolio — without real demo data, the AI assistant looks useless
  • The AI query path uses a separate read-only DB connection (its own least-privilege Postgres role) — never the app's read-write Prisma client
  • (marketing) is the public landing site (operato.app) — static, SEO'd, no auth; see the Landing Page tab

Interview Talking Points

Questions you'll definitely get. Answers that signal senior engineering judgment and product thinking.

"Walk me through the project."
Operato is an AI operating system for small businesses, and I shipped restaurant as the first vertical. The core — multi-tenant model, Better Auth, Razorpay billing, dashboard shell, and an AI engine that does text-to-SQL over the tenant's real data — is vertical-agnostic. The restaurant-specific part is a module pack (menu, orders, tables) plus an AI schema context. So I'm not demoing a CRUD app; I'm demoing one industry, deeply built, on a platform designed to add more.
"How does the multi-tenant architecture work?"
Shared database, shared schema — every table has a restaurantId foreign key. On every API route, I extract restaurantId from the URL, verify the authenticated user (Better Auth session) belongs to that restaurant via RestaurantMember, then filter all queries by that restaurantId — with Postgres Row-Level Security as a backstop. I considered separate schemas per tenant but chose shared because it's simpler operationally and appropriate for early-stage scale. The same tenant column generalises across verticals, and I'd migrate to schema-per-tenant as a natural next step.
"How did you make the AI text-to-SQL safe?"
Defense in depth, because trusting the model is not a strategy. The SQL step uses generateObject with a Zod schema, so I get a typed { sql, params } object — not free text I have to parse. Before execution I reject writes, stacked statements and comments. Execution runs on a dedicated read-only Postgres role inside a read-only transaction with a statement_timeout and a forced LIMIT. And tenant isolation is enforced by Postgres Row-Level Security keyed off the request's restaurantId — so even if Gemini omitted the WHERE clause entirely, the database returns only the authenticated restaurant's rows. The guarantee lives in the database and my code, not in the prompt.
"How does streaming work in the AI responses?"
I use the Vercel AI SDK's streamText on the server, which returns a ReadableStream from Gemini via @ai-sdk/google. On the client, useChat() from the AI SDK handles the SSE connection and renders tokens as they arrive. Users see the first words within ~300ms even when Gemini is generating a long summary. The key architectural choice was separating the SQL generation step (non-streaming, structured JSON) from the natural-language response step (streaming).
"How would you add a second industry — say a café or a retail store?"
It's a config pack, not a fork. The shared core (tenancy, auth, billing, inventory, CRM, AI engine) stays. I add a vertical definition under lib/verticals that declares the domain entities (Catalog instead of Menu, Counters instead of Tables), the dashboard KPIs, and — crucially — the AI schema context plus example question-to-SQL pairs for that domain. Because the AI reads a generated description rather than the raw schema, swapping the context is most of the work. That reuse ratio is the entire point of building it as an OS.
"Why Better Auth over Clerk or NextAuth?"
The deciding factor was data ownership in a multi-tenant Postgres app. With Clerk, users live on Clerk's servers, so I'd need an inbound webhook just to mirror new users into my DB before attaching them to a restaurant — a whole sync layer with signing secrets, retries and ordering races. Better Auth writes the user row directly into my Neon Postgres, so onboarding creates the restaurant and the owner membership in one local transaction, and RestaurantMember is just a foreign key to user.id. The honest trade-off is that I own the auth surface and its maintenance — but I get zero vendor lock-in, no per-MAU pricing, and identity in the same transaction boundary as the rest of my data. If team invitations grow into a real feature, Better Auth's organization plugin adds them without switching providers.
"What would you do differently at real scale?"
Schema-per-tenant instead of shared DB for stronger data isolation. A proper queue (BullMQ or Inngest) for weekly summaries instead of Vercel Cron — Cron doesn't retry failed jobs reliably. Redis caching for common AI queries. Rate limiting on the AI endpoints per tenant per day. And proper observability — right now I log to the AiQuery table, which works for a portfolio, but I'd use something like Datadog in production.
"How did you handle the database context for AI?"
I built a schema-context.ts that generates a natural-language description of the DB — table names, column names, relationships, what each table represents — rather than sending the raw Prisma schema. The Prisma schema has too many technical details the model doesn't need and it wastes tokens. I also inject 5–6 example question-to-SQL pairs that show the model the pattern for common business queries, which significantly improved accuracy on domain-specific questions. This file is also the per-vertical seam.

Build notes

  • Static + fast. RSC, no client data-fetching, target LCP < 1.5s — a landing page that streams a heavy dashboard is a bad first impression
  • SEO. A metadata export per page (title, description, canonical), an OG image, and JSON-LD for the product — this is the page Google indexes
  • One demo CTA that matters. The hero's live chat (or a 90-second Loom) is the single most persuasive asset; everything else supports it
  • Reuses the design system. Same Tailwind tokens + Shadcn primitives as the dashboard, so the brand is consistent from landing → app
  • Timeline: already slotted into August, Week 4 ("Razorpay + Landing page + Demo") — built last, once there's a real product to show
